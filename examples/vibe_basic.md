# Vibe Coding 基础：一小时课程教案

## 一、课程信息

- **课程名称**：Vibe Coding 基础
- **课程时长**：60 分钟
- **适合对象**：
  - 对 AI 编程有兴趣的产品、设计、开发者
  - 刚开始使用 Claude Code / AI Agent 的同学
  - 已经会写一点提示词，但不稳定、不可复现的人
- **课程目标**：
  1. 让学员理解什么是 Vibe Coding，以及它和“随便让 AI 写点代码”的区别。
  2. 让学员掌握一个可执行的协作框架：**RIPER**。
  3. 让学员学会把模糊需求“显化”为 agent 能执行的语言。
  4. 让学员知道如何通过 **statusline、CLAUDE.md、handoff.md** 提高 AI 编程稳定性。
  5. 通过 demo 理解“一键直出”与“逐步协作”的边界。

---

## 二、课程核心结论

本节课希望学员最后记住三句话：

1. **Vibe Coding 不是瞎试，而是用自然语言做软件协作。**
2. **写不好提示词，通常不是表达不够礼貌，而是需求不够显化。**
3. **AI 编程的稳定产出，靠的不是玄学，而是流程：RIPER。**

## 五、详细教案

## 0-5 分钟：开场——什么是 Vibe Coding

### 教学目标

让学员先建立正确预期：Vibe Coding 不是“靠感觉乱写”，而是“用自然语言驱动软件实现”。

### 讲师讲解要点

可以这样开场：

> 过去我们写软件，是人读需求文档、人拆任务、人写代码、人测。现在多了一个很强的协作者：agent。Vibe Coding 的本质，不是把脑子交给 AI，而是把原来写在脑子里的判断、约束、目标，用语言显化出来，让 AI 能接手更多执行工作。

### 板书 / 幻灯关键词

- 自然语言 = 新的协作界面
- 随机出活 ≠ 稳定交付
- AI 写代码的瓶颈，常常不是代码能力，而是需求表达能力

### 互动提问

- 你让 AI 写代码时，最常见的问题是什么？
  - 写偏了
  - 写得能跑但不可维护
  - 改一处坏三处
  - 不理解真实意图

### 讲师过渡语

> 所以这节课不是教大家“怎么问一句神奇提示词”，而是教一个可复用的协作方法。

## 5-12 分钟：环境与 statusline——让 agent 随时知道自己在哪

### 教学目标

让学员理解：AI 编程质量高度依赖上下文可见性，`statusline` 是最低成本、最高收益的增强。

### 讲解内容

#### 1. statusline 是什么

是给 Claude Code 增加一个稳定可见的状态条，让 agent 和人都能一眼看到：

- 当前 **git 分支**
- 当前 **工作路径**
- 当前 **剩余 context window**

#### 2. 为什么这很重要

如果 agent 不知道自己在哪个目录、哪个分支、还剩多少上下文，它就容易：

- 改错目录
- 在错误分支上工作
- 上下文溢出后失忆
- 在长会话中重复探索

#### 3. statusline 的价值

- 降低环境理解成本
- 提高多轮协作稳定性
- 提醒何时该总结、写 `handoff.md`
- 让“一键直出”更接近“有边界的直出”

### 演示建议

现场展示一个带 statusline 的终端界面，指出三个区域：

- `branch: feature/export-ui`
- `cwd: ~/projects/uigen`
- `context left: 32%`

然后说：

> 你把这些状态显化出来，相当于给 agent 加了仪表盘。AI 不是更聪明了，但它更不容易迷路了。

### 板书关键词

- 分支
- 路径
- 剩余上下文
- 可见性

### 小结句

> 先别急着写提示词，先把工作现场布置好。

## 12-22 分钟：显化你的需求——从“会意”变“指令”

### 教学目标

让学员掌握把隐晦表达翻译成直接需求的方法，这是 Vibe Coding 的核心能力。

### 切入比喻

使用用户提供的饭桌例子：

> 如果有人阴阳你，别解释、别否认、别顺着他说。最有效的方法，是把他的潜台词翻译成大白话。
>
> 比如二舅在年夜饭桌上说：“xxx 现在是大人物了，我们这帮亲戚高攀不起了。”
>
> 你如果回：“没有没有，怎么会呢。”其实已经掉进对方的话术里了。
>
> 更好的方式是直接翻译：
>
> **“二舅，您是说我现在混得好了，忘本了，看不起您老了？”**
>
> 这时对方往往会退回去，因为潜台词一旦被显化，就不能再模糊施压了。

### 引申到 agent 协作

指挥 agent 也是一样。

很多人给 AI 的需求其实是“带潜台词的、靠猜的、期待会意的表达”。

比如：

- “帮我把这个页面做高级一点。”
- “这个功能体验不好，你优化下。”
- “做一个像 base44 那样的东西。”
- “一键直出一个监控系统。”

这些话对人来说都不够清楚，对 agent 来说更不清楚。

### 显化公式

可以给学员一个简单公式：

> **模糊愿望 = 场景 + 目标 + 约束 + 验收**

把一句模糊话，拆成四部分：

1. **场景**：现在是什么情况？
2. **目标**：你到底想达到什么结果？
3. **约束**：不能做什么、必须遵守什么？
4. **验收**：怎样算做完？

### 示例 1：页面做高级一点

#### 模糊表达

> 帮我把这个页面做高级一点。

#### 显化后

> 这是一个 SaaS 后台首页。目标是让它看起来更像可收费产品，而不是 demo。请优先优化信息层级、留白、配色和按钮样式，不改变现有功能结构，不引入新组件库。验收标准：首屏视觉重点清晰、主操作按钮突出、卡片间距统一、整体风格偏 B2B 工具感。

### 示例 2：做一个像 base44 那样的东西

#### 模糊表达

> 做一个像 base44 那样的东西。

#### 显化后

> 我想做一个简化版的前端生成工具。用户输入自然语言后，系统生成前端组件代码并实时渲染。第一版只支持：输入框、按钮、卡片、表单这四类组件；使用 React；本地运行；暂不做登录、数据库、多用户。验收标准：用户输入描述后，能看到生成代码和预览结果，并可再次编辑迭代。

### 课堂练习（2 分钟）

让学员把这句话显化：

> “帮我做一个 BTC 价格监控。”

讲师可引导出一个参考答案：

- 场景：命令行工具/网页小工具
- 目标：实时查看 BTC 最新价格，并在超过阈值时提醒
- 约束：先做本地版本，不接交易、不存账户、不做复杂 UI
- 验收：能每隔 N 秒刷新一次，价格突破阈值时输出提醒

### 小结句

> AI 最怕的不是需求难，而是需求虚。你越显化，agent 越稳定。

## 22-35 分钟：思维框架——RIPER

### 教学目标

给学员一套完整工作流，让他们知道什么时候探索、什么时候定稿、什么时候执行、什么时候回头审查。

### 讲解方式

从传统软件公司开发流程类比切入：

> 在传统软件公司里，几乎没有哪个成熟团队会让工程师听一句“你看着做”就直接开写。通常要经历：需求澄清、方案比较、计划制定、编码实现、测试验收、代码评审。RIPER 只是把这套流程，翻译成你和 agent 的协作语言。

### RIPER 展开

## R — Research

### 定义

给模型一个愿望，先问“怎么实现”，而不是立刻让它开写。

### 作用

- 快速摸清实现路径
- 发现未知约束
- 暴露隐藏难点
- 避免一上来就写错方向

### 示例提示

> 我想做一个 BTC 价格监控工具，请先不要写代码，先研究有哪些实现方案：命令行、网页、桌面通知各自的优缺点是什么？所需依赖、数据源、实时性、开发复杂度分别如何？

### 常见错误

- 刚有想法就直接“帮我写完”
- 没搞清依赖和边界就开工

---

## I — Innovation

### 定义

把需求拆成多个环节，讨论每个环节的实现选项。

### 作用

不是让模型“创意发散”，而是系统比较方案。

### 应该讨论的内容

- 数据源怎么取？
- 架构怎么分层？
- 用轮询还是 websocket？
- 阈值告警做终端输出还是系统通知？
- 单文件脚本还是小型项目结构？

### 示例提示

> 把这个工具拆成数据获取、状态管理、告警策略、展示方式四部分。每一部分给我 2-3 个实现选项，并说明取舍，不要直接下结论。

### 讲师强调

> 很多人跳过 Innovation，所以最后不是“没有方案”，而是“只有 agent 替你偷偷做出的方案”。

---

## P — Plan

### 定义

当现状、动机、实现过程、验证过程都清楚时，定稿成一个执行计划。

### 判断“何时可以敲定”的标准

当以下四件事都清楚时：

1. **现状**：我现在手上有什么
2. **动机**：我为什么做这个版本
3. **实现过程**：分几步完成
4. **验证过程**：怎么判断做对了

### 计划里至少应包含

- 功能范围
- 技术选型
- 文件结构
- 开发顺序
- 测试策略
- 验收标准

### 示例提示

> 基于刚才讨论，输出一个最小可执行计划。要求分阶段、列文件、列测试、列验收标准。确认前不要写代码。

### 常见错误

- 计划只有功能，没有验证
- 计划写成愿望清单，没有执行顺序

---

## E — Execute

### 定义

按计划一步步实现，**先写测试，再写代码，直到测试通过**。

### 关键原则

- 一次只做一步
- 每一步都可验证
- 不要让 agent 自己无限扩写

### 重要补充：做不完先存 `handoff.md`

比起直接压缩上下文，`handoff.md` 更可控，因为它是可编辑、可审阅、可继承的外部记忆。

建议 `handoff.md` 记录：

- 当前做到哪一步
- 已完成哪些文件修改
- 哪些决策已敲定
- 哪些问题未解决
- 下一步具体做什么

### 讲师金句

> compact 是失忆前的自救，`handoff.md` 是可管理的项目记忆。

---

## R — Review

### 定义

让模型对照计划和设计，审查代码与原计划的偏移程度。

### Review 不是“再看一遍代码”

而是检查：

- 是否偏离原需求
- 是否偷偷加了没讨论过的复杂度
- 是否漏了验收项
- 测试是否覆盖关键路径
- 文档是否足以交接

### 示例提示

> 请对照计划文档审查当前实现，指出与原设计的偏移、遗漏的测试、隐藏的复杂度，以及下一轮应收敛的问题。

### 小结句

> 没有 Review 的 Vibe Coding，往往前面越顺，后面越难维护。

## 35-48 分钟：Demo 1——BTC 价格监控

### 教学目标

通过一个简单、清晰、能短时间做完的案例，演示 RIPER 不是理论，而是实际可操作的节奏。

### Demo 目标

做一个“一键直出”的 BTC 价格监控最小版本：

- 获取 BTC 当前价格
- 支持设定阈值
- 到达阈值时提醒
- 先做本地可运行版本

### 演示节奏

#### 第一步：Research（2 分钟）

向 agent 询问：

- 用什么 API 获取 BTC 价格
- 轮询还是 websocket
- CLI 还是网页最适合课堂演示

讲师点评：

> 课堂场景要稳，优先选依赖少、反馈快、失败面小的方案。所以可能更适合先做命令行版本。

#### 第二步：Innovation（2 分钟）

讨论几种实现选择：

- 数据源：公开 API
- 展示：命令行刷新 / 简单网页
- 告警：文字提醒 / 声音 / 系统通知

讲师现场做取舍：

> 第一版选“公开 API + 命令行输出 + 阈值提醒”，因为最小闭环最清晰。

#### 第三步：Plan（3 分钟）

让 agent 输出一个最小计划：

- `fetch_price`
- `check_threshold`
- `main loop`
- 简单测试
- README 使用方式

#### 第四步：Execute（4 分钟）

让 agent 按步骤实现：

1. 先写测试或至少列出验证方法
2. 写获取价格函数
3. 写阈值判断
4. 写主循环
5. 运行验证

#### 第五步：Review（2 分钟）

让 agent 自查：

- 是否处理 API 失败
- 是否处理用户输入异常
- 是否把配置写死了
- 是否便于下一步扩展到网页版

### 讲师点评重点

- “一键直出”可以，但前提是边界小
- demo 越小，越适合完整走 RIPER
- 真正的能力不是让 AI 一次写 500 行，而是让它始终不偏航

## 48-55 分钟：Demo 2 / 3——从小工具到项目协作

这一段不必完整展开实现，重点是让学员看到同一套方法如何迁移到不同项目类型。

## Demo 2：slidet

### 项目一句话介绍

一个终端版本的 PPT，读取 Markdown 并渲染成演示内容。

### 讲解重点

这个案例适合说明：

- AI 不只会写 CRUD，也能处理“工具型产品”
- Markdown -> 解析 -> 渲染 -> 交互，是典型的多环节需求
- 如果不先拆模块，agent 很容易把解析、渲染、快捷键逻辑搅在一起

### 可强调的 RIPER 应用

- Research：终端渲染有哪些库可选？
- Innovation：分页模型、主题系统、快捷键设计怎么选？
- Plan：先支持标题页和普通页，还是一步支持全部 Markdown 语法？
- Execute：先做最小渲染闭环
- Review：检查是否过早抽象

## Demo 3：uigen

### 项目一句话介绍

一个简化版的 base44，支持前端组件 vibe coding 和渲染。

### 讲解重点

这个案例适合说明：

- 当项目开始复杂化，必须把探索结果沉淀下来
- **`CLAUDE.md`** 是项目级记忆，不要每轮重讲一遍背景
- 可以在课程直播里演示：**为它增加一个“前端导出功能”**

### 重点提示

#### `CLAUDE.md` 的作用

把模型已经探索清楚的结果写成文件，例如：

- 项目目标
- 技术栈
- 目录规范
- 组件生成约束
- 代码风格
- 已知坑点

这样下次继续时，只需要让模型先读 `CLAUDE.md`，而不是重新探索。

#### 可给学员的结论

> 小项目靠上下文记忆，大项目靠文件化记忆。

---

## 55-60 分钟：总结与答疑

### 结课总结

回收全课，重复三层结构：

#### 1. 基础设施层

- `statusline`
- `CLAUDE.md`
- `handoff.md`

它们解决的是：**上下文可见、项目记忆、跨轮交接**。

#### 2. 表达层

- 显化需求

它解决的是：**让 agent 不猜。**

#### 3. 流程层

- RIPER

它解决的是：**让产出可控、可复现、可审查。**

### 结课金句

可以用这一句收尾：

> Vibe Coding 最强的地方，不是“你一句话它就全做了”，而是“你终于可以把做软件这件事，变成一场高带宽协作”。

### 课后练习

给学员一个练习题：

> 任选一个你最近想做的小工具，用 RIPER 写出：
> 1. 一段显化后的需求描述
> 2. 一个最小执行计划
> 3. 一个 `handoff.md` 模板

## 六、讲师备注：易错点与纠偏

### 易错点 1：把 Vibe Coding 讲成“万能自动编程”

纠偏：强调它是协作增强，不是责任外包。

### 易错点 2：只讲提示词技巧，不讲工程流程

纠偏：提示词只是接口，流程才决定稳定性。

### 易错点 3：demo 太大

纠偏：一小时课程里，完整闭环比炫技更重要。

### 易错点 4：把 `CLAUDE.md` 和 `handoff.md` 混用

纠偏：

- `CLAUDE.md`：长期项目规则与共识
- `handoff.md`：本轮进度与交接记录

## 七、可直接投屏的一页版总结

## Vibe Coding = 基础设施 + 显化表达 + 协作流程

- **基础设施**：statusline / CLAUDE.md / handoff.md
- **显化表达**：把模糊愿望翻译成场景、目标、约束、验收
- **协作流程**：RIPER
  - Research
  - Innovation
  - Plan
  - Execute
  - Review

### 一句话定义

> 让 AI 稳定产出的关键，不是更会“许愿”，而是更会“定义工作”。

## 八、附：可口述的 RIPER 速记版

> 先研究，再比较；
> 方案定了，写计划；
> 按计划做，先测后码；
> 做完回看，检查偏差。

## 九、附：课堂上可直接使用的提示模板

### 1. Research 模板

```text
我想做一个【项目/功能】。先不要写代码。
请先帮我研究：有哪些可行实现方案？
分别需要什么依赖、复杂度如何、风险点是什么？
最后给出推荐方案和理由。
```

### 2. Innovation 模板

```text
把这个需求拆成几个关键环节。
针对每个环节给出 2-3 个实现选项，分析优缺点与适用场景。
先不要写代码，也不要直接进入实现。
```

### 3. Plan 模板

```text
基于刚才确定的方案，输出一个最小可执行计划。
要求包含：目标、范围、技术选型、文件结构、开发顺序、测试方式、验收标准。
确认前不要开始编码。
```

### 4. Execute 模板

```text
按计划逐步实现，一次只完成一步。
优先先写测试或验证方法，再写代码。
每完成一步就说明你做了什么、如何验证、下一步是什么。
```

### 5. Review 模板

```text
请对照原计划审查当前实现。
指出：偏离计划的地方、遗漏的测试、隐藏复杂度、文档缺口，以及建议的下一步收敛方向。
```
