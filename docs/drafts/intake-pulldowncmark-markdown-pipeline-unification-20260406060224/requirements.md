# Requirements: Unify Markdown Pipeline on pulldown-cmark

## 1. Problem Statement
`slidet` 当前在 Markdown 处理上存在分裂路径：`src/markdown.rs` 使用 `pulldown-cmark` 做块解析和标题提取，但 `src/ui.rs` 仍通过 `tui-markdown::from_str` 进行文本渲染。用户希望把 Markdown 能力统一到 `pulldown-cmark`，避免多套语义来源导致行为不一致、维护成本升高，以及未来扩展常见 Markdown 元素时边界不清。

## 2. Desired Outcome
项目应以 `pulldown-cmark` 作为唯一 Markdown 管线来源，既负责解析语义，也驱动终端展示逻辑；同时在不打破现有 TUI 架构、交互模式和 slide 加载约定的前提下，提升常见 Markdown 元素在终端中的显示效果与一致性。

## 3. Constraints
- **Tech stack**: 继续基于 Rust、`ratatui`/`crossterm` 现有架构开发，Markdown 管线统一到 `pulldown-cmark`。
- **Hard boundaries**: 尽量保留当前 `Browse` / `Present` 交互模型、按键约定、目录扫描与按文件名字典序加载 slide 的机制；优先在现有模块边界内演进，不将 loader、parser、UI、状态管理逻辑混杂重构。

## 4. Scope
### In Scope
- 用 `pulldown-cmark` 统一 `slidet` 的 Markdown 解析与渲染语义来源。
- 清理或替换当前依赖 `tui-markdown` 的文本渲染路径。
- 改善常见 Markdown 元素在终端中的展示效果，至少覆盖标题、列表、引用、代码块、表格、图片相关文本表现。
- 复用或补充现有样例与测试，尤其是 `examples/04-*`、`examples/05-*` 作为回归输入。

### Out of Scope
- 重做整体 TUI 信息架构或交互模型。
- 改变 slide 文件组织方式或加载顺序约定。
- 超出终端能力范围的富媒体渲染设计。

## 5. Acceptance Criteria
- [ ] `cargo check` 与 `cargo test` 通过。
- [ ] 示例 slides 中常见 Markdown 元素能够稳定显示。
- [ ] Markdown 显示行为围绕 `pulldown-cmark` 统一，而不是依赖分裂的多套 Markdown 语义路径。
