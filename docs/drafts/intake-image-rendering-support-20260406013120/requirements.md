# Requirements: Markdown 幻灯片图片显示

## 1. Problem Statement
`slidet` 当前会把 Markdown 图片块解析为 `Image`，但 UI 层并未真正渲染图片，而是输出类似 `[image render] <path>` 的占位文本。对于终端 Markdown 幻灯片播放器来说，这意味着包含截图、流程图或插图的 slide 无法按预期展示，演示价值明显受限。

当前代码中 `src/image.rs` 已具备基础文件存在性检查与终端图片能力探测，项目也已依赖 `image` 与 `ratatui-image`。因此问题不是“是否识别到图片”，而是“如何在现有架构内把 jpg/png 打通为首版真渲染，同时保持 svg 和异常场景的可读 fallback”。

## 2. Desired Outcome
用户把包含 `![alt](path/to/file.jpg)` 或 `![alt](path/to/file.png)` 的 Markdown slide 目录传给 `slidet` 时，在支持图片显示的终端中，`Browse` 与 `Present` 模式都能看到实际图片内容，而不是路径占位文本。

当图片为 `svg` 时，系统不尝试首版真渲染，而是输出清晰的文本提示，明确说明该资源已找到，但 `svg` 当前暂不支持渲染。若文件缺失或终端不支持图片，也必须继续保留现有的 graceful fallback，不得 panic。

## 3. Constraints
- **Tech stack**: 继续使用 Rust、`ratatui`、`crossterm`、`pulldown-cmark` 和现有 `image.rs` / `ui.rs` / `markdown.rs` 模块边界实现，不引入破坏现有结构的跨层耦合。
- **Hard boundaries**: 首版只承诺 `jpg/png` 真渲染；`svg` 仅提供明确 fallback；缺失文件和终端能力不足时必须维持可读降级；不要破坏“按目录加载 `.md` slides”的既有约定；优先补测试再修改实现。

## 4. Scope
### In Scope
- 在现有 Markdown 图片块模型上支持 `jpg` / `jpeg` / `png` 资源的真实图片渲染链路。
- 对 `svg` 资源输出明确占位文本，文案中包含资源路径和“svg 暂不支持渲染”的含义。
- 保留并验证缺失图片、终端不支持图片时的 fallback 行为。
- 为图片渲染成功、`svg` fallback、缺失文件 fallback 增补或调整自动化测试。
- 复用或补充示例数据，保证图片相关行为可回归验证。

### Out of Scope
- `svg` 的首版真渲染、栅格化、缩放优化与复杂兼容性处理。
- 新增 Markdown 图片语法之外的媒体类型支持。
- 对图片布局策略做大幅视觉重构，例如复杂对齐、居中算法或高级缩放规则。
- 终端能力探测机制的大规模重写。

## 5. Acceptance Criteria
- [ ] Markdown 中的 `![alt](file.jpg)` 与 `![alt](file.png)` 在支持图片显示的终端中可实际显示为图片内容。
- [ ] `Browse` 模式与 `Present` 模式都遵循相同的图片加载与降级规则，不再只输出 `[image render] <path>`。
- [ ] `svg` 资源在首版显示为明确文本占位，能让用户分辨“文件存在但格式暂不支持渲染”。
- [ ] 图片文件缺失时继续显示缺失提示；终端不支持图片时继续显示能力不足提示；两种情况都不得 panic。
- [ ] 自动化测试覆盖至少三类情况：`jpg/png` 可渲染链路、`svg` fallback、缺失文件 fallback。
